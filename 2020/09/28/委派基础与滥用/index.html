委派基础与滥用（笔记）

发表于 2020-09-28
本章主要介绍委派的基础与滥用

委派简介

Kerberos 是一种身份验证协议，使得用户可以使用由域控制器发放的票据向服务证明其身份；

而 Kerberos 委派用于多层应用程序/服务的情况，服务可代表用户访问另一个服务。

委派的作用：通过委派，资源服务可以知道是谁在访问它，以及是否有权限访问。

委派的案例

用户访问 Web 应用程序，Web 应用程序需要调用资源数据库中的数据，此处有三个角色：1）浏览网站的用户；2）Web 应用程序（运行在 Web 服务帐户上）；3）资源数据库。

委派的实际案例

1）用户访问 Web 应用程序，需要进行身份验证；

2）由于 Web 应用程序需要调用数据库中的数据，所以，数据库需要为 Web 服务账户授予足够的访问权限；

3）由于数据库需要知道客户端用户是否具有访问数据的权限，所以我们需要为 Web 服务帐户配置委派，让其可以代表客户端用户访问数据库，这样数据库便可以判断客户端用户是否具备访问权限。

如果没有委派，数据库将无法知道哪个用户在访问其中数据，因为在之前数据库已经授予了 Web 服务账户具有足够的访问权限。

备注：票据中包含用户的 SID 信息。

用户 UserAccountControl 属性

所有的活动目录安全主体（对象）都具有 UserAccountControl 属性。这个属性是一个位集，这意味着每个二进制数字都有一个特定的含义。这些位值在 Windows SDK 的 lmaccess.h 中定义。关于委派的有以下几个：

Not_Delegation（敏感账户，不能被委派）

KDC 通过该标志判断用户是否允许被委派。

Trusted_For_Delegation（不受限委派）

KDC 通过该标志判断是否应用不受限委派（是否将用户的 TGT 副本放入返回的 TGS 中）。

Trusted_To_Auth_For_Delegation（T2A4D）（受限委派-启用协议转换）

在 S4U2Self 中，KDC 通过该标志判断是否应用受限委派并启用协议转换（是否返回可转发 TGS ）。

注意：用户账户的 UserAccountControl 属性与 Windows UAC 机制不同，它设置活动目录总账户的特定属性，如：账户是否被禁言、锁定，密码是否会过期等等。

用户属性

msDs-AllowedToDelegateTo（受限委派-仅允许委派到哪些服务）

在 S4U2Proxy 中，KDC 通过该属性判断是否应用受限委派，以及服务账户允许代表用户访问哪些服务。

msDs-AllowedToActOnBehalfOfOtherIdentity（基于资源的受限委派-哪些账户可以代表其他用户访问我）

在 S4U2Proxy 中，KDC 通过该属性判断是否应用基于资源的受限委派，以及哪些账户能访问目标服务。

不受限委派

Windows Server 2000 中引入不受限委派，是 Windows Server 2000 中委派的唯一选项，出于向后兼容原因，后续版本也保留了该功能

配置：域管理员可以在 用户/计算机 属性的委派选项卡选择 “信任此用户/计算机来委派任何服务” 进行配置。

作用：该类型委派允许服务账户可以代表用户访问任意服务。

Trusted_For_Delegation

  • 当一个账户配置为 “信任此计算机/用户来委派任何服务” 时，该账户的 UserAccountControl 属性中将包含此字段
  • KDC 通过该字段判断服务是否配置为 “不受限委派”，并决定是否在（TGS_REP）返回的 TGS 中包含用户的 TGT
  • 修改用户 UserAccountControl 属性中将包含此字段需要具备 SeEnableDelegationPrivilege 权限

不受限委派的流程

1）TGS_REQ：用户向 KDC 发送 TGT 请求服务的 TGS；

2）KDC 判断：服务账户 UAC 属性中是否含有 TrustedForDelegation 标志（如果是）

3）TGS_REP：KDC 把包含用户 TGT 的 TGS 返回给用户

4）AP_REQ：用户使用该 TGS 访问服务

服务账户在接收到该 TGS 后，获取到其中的用户 TGT，将用户 TGT 放入 LSASS 中存储供后续使用；

后续，服务账户可以使用用户的 TGT 代表该用户向 KDC 请求其他任意服务的 TGS。

受限委派

为解决不受限委派所存在的安全问题，Windows Server 2003 中引入受限委派，该委派类型包括一组称为 S4U2Self 和 S4U2Proxy 的 Kerberos 协议扩展，这些扩展还支持 “协议转换” 功能。

配置：域管理员可以在 用户/计算机 属性的委派选项卡选择 “仅信任此用户/计算机来委派指定的服务” 进行配置，并通过添加服务主体名称（SPN）来指定允许访问的服务。

作用：该类型委派仅允许服务账户可以代表用户访问指定服务。

用户属性 msDs-AllowedToDelegateTo

  • 这是服务账户（通常是计算机或用户账户）对象的属性
  • 它包含一个服务主体名称（SPN）列表，表示账户可以将用户 token 转发给哪些指定的 kerberos 服务
  • 修改该属性需要在域控中具有 SeEnableDelegationPrivilege 权限

备注：如果域林中所有域控制器均为 Windows Server 2012 及以后版本，该类型的委派可以跨域。

基于资源的受限委派

Windows Server 2012 中引入基于资源的受限委派。该类型委派中，目标服务可以指定哪些账户可代表其他用户访问自身。

配置：账户可以根据需要，编辑可代表其他用户访问自身的账户列表，此列表用于基于资源的受限委派。

注意：它的配置方向与传统受限委派刚好相反，传统受限委派是允许服务账户代表用户访问那些目标服务，而基于资源的受限委派是目标服务允许哪些账户可以代表用户访问自身。

用户属性 msDs-AllowedToActOnBehalfOfOtherIdentity

  • 这是服务账户（通常是计算机或用户账户）对象的属性
  • 该属性包含一个账户列表，以确定请求者是否能够代表其他用户身份访问自身账户所运行的服务
  • 通常在后端资源数据库的服务账户上进行配置，指定哪些中间层服务可以代表用户向 KDC 请求访问数据库服务的票据

S4U2Self（ServiceForUserToSelf）

该扩展是为了允许服务委派不支持 kerberos 认证的用户，也就是说，服务可以在不从客户端用户处获得 TGS 的情况下，代表（已通过其他方式经过认证的）用户向 KDC 请求一个用于访问自身的 TGS。

简单来说：用户通过其他方式（非 kerberos 的方式）对服务进行认证，服务仍然可以代表该用户向 KDC 请求访问自身的 TGS。

S4U2Self 流程

用户已通过非 Kerberos 方式对服务进行身份验证（例如：已通过 NTLM 进行认证 ）

1）TGS_REQ：服务账户使用自身 TGT 代表用户向 KDC 请求一个访问自身的 TGS

2）KDC 判断：1. 服务账户是否为服务的所有者（是）；2. 服务账户 UAC 属性是否含有 TrustedToAuthForDelegation 标志（是）

3）TGS_REP：KDC 向服务返回一个可转发的 TGS（该 TGS 可以在 S4U2Proxy 中使用）

解释：

第一步中：一个用户可以为其他任意用户执行 S4U2Self 过程，需要提供用户的名称，不需要提供该用户的密码。

第三步中：如果该账户没有服务，KDC 将不返回 TGS （所以只有设置了 SPN 的账户才能执行 S4U2Self）。请求的账户 userAccountControl 属性中如果包含 TrustedToAuthForDelegation 标志，KDC 将返回可转发 TGS，如果没有该标志，那么 KDC 将返回 不可转发 TGS。

第四步中：如果用户不允许被委派（敏感账户或受保护用户组成员），KDC 将返回一个不可转发 TGS。

Trusted_To_Auth_For_Delegation（T2A4D）

当账户配置为 “仅信任此计算机来委派指定的服务 - 使用任何身份验证协议” 时，该账户的 UserAccountControl 属性中将包含该字段。

当账户配置为 “仅信任此计算机来委派指定的服务 - 仅 kerberos ” 时，该账户的 UserAccountControl 属性中将不包含该字段。

作用：该字段表明为受限委派 “启用协议转换”，允许用户通过非 kerberos 方式对服务账户进行认证的情况下，服务账户能够执行 S4U2Self，代表用户向 KDC 请求一个访问自身的 TGS。

协议转换：用户使用非 kerberos 方式进行认证，执行 S4U2Self 后，服务可获得了用户身份的 票据。

备注：修改 Trusted_To_Auth_For_Delegation 字段需要具备 SeEnableDelegationPrivilege 特权。

S4U2Proxy（ServiceForUserToProxy）

该扩展允许服务账户使用（用户身份的） TGS 代表用户向 KDC 请求访问目标服务的 TGS。

S4U2Proxy 流程

用户将 TGS 发送给服务，或者服务通过 S4U2Self 获得用户身份的 TGS

1）TGS_REQ：服务账户使用用户身份的 TGS 代表用户向 KDC 请求目标服务的 TGS

2-1）KDC 判断：1. 服务账户 msDs-AllowedToDelegateTo 属性的 SPN 列表中是否有目标服务 SPN（是）。2. TGS 是否是可转发的（是，则应用受限委派）

2-2）KDC 判断：1. 服务账户 msDs-AllowedToDelegateTo 属性的 SPN 列表中是否有目标服务 SPN（否，不应用受限委派）。2. 目标服务账户 msDS-AllowedToActOnBehalfOfOtherIdentity 属性的账户列表中是否有服务账户（是，则应用基于资源的受限委派）

3）TGS_REP：KDC 向服务返回可转发 TGS （用户身份）

4）AP_REQ：服务账户使用该 TGS 代表用户访问目标服务

解释：

第二步中：受限委派优先于基于资源的受限委派进行判断；因此如果在第一种情况中，如果 TGS 没有可转发标志，则失败；另外，第二种情况中，基于资源的受限委派不需要可转发标志；如果以上两种情况都为否，则失败。

第三步中：S4U2Proxy 总是返回可转发 TGS。

基础回顾

服务账户配置为 ”信任此用户/计算机来委派任何服务“，服务账户 UAC 属性中含有 TrustedForDelegation 标志

服务账户配置为 “仅信任此用户/计算机来委派指定的服务”，服务账户 msDS-AllowedToDelegateTo 属性包含能访问的 SPN 列表

服务账户配置为 “仅信任此用户/计算机来委派指定的服务 - 仅 kerberos”，服务账户 UAC 属性中没有 T2A4D 标志

服务账户配置为 “仅信任此用户/计算机来委派指定的服务 - 任何身份验证协议”，服务账户 UAC 属性含有 T2A4D 标志，启用 “协议转换”

  • KDC 会检查是否有该标志，如果没有则返回不可转发 TGS，如果有则返回可转发 TGS

S4U2Proxy

服务账户 msDS-AllowedToDelegateTo 属性的 SPN 列表中是否具有 目标服务 的 SPN：

  • 如果收到的 TGS 设置了可转发标志，KDC 将返回一个可转发 TGS，并应用受限委派
  • 如果收到的 TGS 没设置可转发标志，则请求失败

服务账户 msDs-AllowedToActOnBehalfOfOtherIdentity 属性的 账户列表中是否存在 请求方账户，判断是否应用 “基于资源的受限委派”

  • 无论收到的 TGS 是否设置可转发标志，都应用基于资源的受限委派，并返回一个可转发 TGS

不受限委派的滥用

滥用不受限委派的主要因素在于能获得认证用户的 TGT，并将其重用。

滥用 不受限委派 结合 打印机漏洞，实现 RCE

打印机漏洞简介：通过 PRC 调用 RpcRemoteFindFirstPrinterChangeNotification（Opnum 62）来欺骗受害主机的打印机后台程序通过 SMB 连接我们指定的 IP/主机名

前提：我们已经控制配置了 “不受限委派” 的 计算机/用户

利用流程

0）我们可以先通过 powershell Get-ADComputer 查看该账户是否配置了 “不受限委派”

1）使用 Rubeus 的 monitor 模块（该模块在滥用不受限委派中非常有用）监听本地新登录事件，并获取 TGT

2）运行 SpoolSample 或 dementor.py 操作目标主机的打印机后台程序访问我们的已控机器

3）我们使用的从监听处获得的 TGT 代表本地管理员执行 S4U 请求访问目标主机的 TGS

4）通过 pass the ticket 的方式将获得的 TGS 注入到当前会话中，并访问目标主机

受限委派的滥用

滥用 受限委派

前提：我们已控制 配置了 “受限委派” 的计算机/用户账户，以及该账户的密码明文或 hash

利用流程

1）通过 powerview 查看该账户 msDs-AllowedToDelegateTo 属性的值

2）使用该账户的凭证执行 asktgt，请求一个 TGT

3）通过上一步获得的 TGT 执行 S4U，代表管理员请求一个访问目标服务的 TGS

4）通过 pass the ticket 的方式将获得的 TGS 注入到当前会话中，便可访问目标

滥用 用户权限 SeEnableDelegationPrivilege

SeEnableDelegationPrivilege 权限简介：默认情况下，只有域管理员（或类似级别）具备该权限。该权限可以修改其他账户 UAC 属性中 Trusted_For_Delegation 和 Trusted_For_Auth_To_Delegation 字段，以及其他账户 msDS-AllowedToDelegateTo 属性的值。

前提：我们已经控制具备该权限的账户，并且该账户对其他域用户具备 GenericAll / GenericWrite 等权限；以及另一个已控的域用户X

利用流程

1）通过域管账户修改 用户X 的 msDS-AllowedToDelegateTo 属性的值，再其中添加想要访问的任何目标服务

2）使用 用户X 的凭证 执行 asktgt 请求一个 TGT

3）使用上一步获得的 TGT 执行 S4U，代表管理员请求一个访问目标服务的 TGS

4）通过 pass the ticket 的方式将获得的 TGS 注入到当前会话中，并访问目标

备注：如果添加的目标服务为 LDAP，我们可通过访问 LDAP 服务修改我们的 ACL，以具备 DCSync 的权限

基于资源的受限委派的滥用

由于修改目标对象的 msds-allowedtoactonbehalfofotheridentity 属性需要对目标对象具备 GenricAll、GenricWrite、WriteDACL 等权限，所以我们可以基于 ACL 修改目标对象的 msds-allowedtoactonbehalfofotheridentity 属性的值，在其中添加我们已控的账户，使我们对目标具有访问权限。

备注：如果我们已控的账户有 T2A4D 标志，那么执行 S4U2Self 获得的 TGS 是可转发；如果我们已控的账户没有 T2A4D（例如：通过滥用 MachineAccountQuota 创建的账户），那么执行 S4U2Self 获得的 TGS 是不可转发的，但是 基于资源的受限委派是可以接收不可转发的 TGS的；并且 S4U2Proxy 总是访问一个 可转发的 TGS。

基于 ACL 的滥用：对目标对象具有相应的DACL（如 GenericWrite 等）是发起基于 ACL 利用的最重要的条件；如果我们不具备，可尝试通过 NTLM 重放攻击，将 NTLM 认证信息重放到 LDAP 服务器上，以便让我们具有相应的 ACL ，对目标对象配置基于资源的受限委派。

基于 ACL，滥用 基于资源的受限委派，实现计算机对象的接管

前提：

1）我们已经控制一个具备相关 DACL 的计算机/用户，它对目标计算机对象具备 GenricAll、GenricWrite、WriteDACL 等权限，允许我们修改/编辑 目标计算机对象的 msds-allowedtoactonbehalfofotheridentity 安全标识符；

2）我们已经控制一个具有 T2A4D 标志的账户；

利用流程

0）在实施利用前，我们需要确认一下 已控账户对目标对象具备相应的 DACL（GenericWrite）等权限，以及具有 T2A4D 标志的账户。

1）将具有 T2A4D 标志的账户添加到 目标对象 msds-allowedtoactonbehalfofotheridentity 属性的 列表中。

2）我们使用具有 T2A4D 标志的账户 代表管理员 执行 S4U 请求目标对象的 TGS。

3）通过 pass the ticket 的方式将票据注入到当前会话中，并访问目标。

清理痕迹：完成利用之后，需要重置目标对象的 msds-allowedtoactonbehalfofotheridentity 安全描述符。

基于 ACL，滥用 基于资源的受限委派，实现计算机对象的接管（绕过 T2A4D 的限制）

前提：

1）与上一个利用方式一样，我们需要控制一个对目标对象具备相关 ACL 的账户；

2）我们通过滥用 MachineAccountQuota 创建一个计算机账户，并设置默认 SPN；该账户没有 T2A4D 标志。

解释：这里我们不需要控制一个具备 T2A4D 标志的账户，而是自创建一个具备 SPN 的计算机账户来执行 S4U；虽然该账户没有 T2A4D 标志，S4U2Self 返回的 TGS 是不可转发的，但是基于资源的受限委派是可以接收的（之前多次提到过）。

利用流程

0）先使用 powerview Get-DomainObjectACL 确定我们已控账户对目标对象具备相关的 ACL；

1）使用 powermad New-MachineAccount 创建一个计算机对象，并设置默认的 SPN；

2）将创建的计算机对象添加到 目标对象 msds-allowedtoactonbehalfofotheridentity 属性的 账户列表中（最后确认下是否成功添加）

3）通过创建的对象 代表管理员执行 S4U，获取访问目标对象的 TGS

4）通过 pass the ticket 的方式将票据注入到当前会话中，并访问目标。

备注：域中的 MachineAccountQuota 默认设置为10，这意味着普通域用户可以最多创建10个新的计算机账户。

基于 ACL，滥用 基于资源的受限委派，实现计算机对象的接管（跳过 S4U2Self）

前提：

1）与上两个利用方式一样，我们需要控制一个对目标对象具备相关 ACL 的账户；

2）我们需要通过社工或水坑等攻击方式来获得用户的 TGS，从而跳过 S4U2Self 步骤，直接执行 S4U2Proxy。

利用流程

1）我们通过各种攻击方式，使受害者访问我们的已控主机（例如：cifs、http）

2）使用 mimikatz 或 Rubeus 的票据导出模块，导出受害者的 TGS

3）将已控账户添加到 目标对象 msds-allowedtoactonbehalfofotheridentity 属性的 账户列表中

4）使用受害者的 TGS 执行 S4U2Proxy，请求访问目标对象的 TGS

5）通过 pass the ticket 的方式将票据注入到当前会话中，并访问目标。

反射型 基于资源的受限委派（类似白银票据）

白银票据简介

银票：在获得服务账户的密码 Hash 的情况下，伪造仅能访问该服务自身的高权限 TGS

银票存在的问题：没有有效 KDC 签名的 PAC，所以在 KDC 校验 PAC 签名的情况下，银票失效

反射型 基于资源的受限委派

默认下，所有的服务都有一个访问控制条目（ACE），允许它们配置能访问自身的账户，也就是基于资源的受限委派；所以，如果我们获取了一个服务账户的凭证，可以配置基于资源的受限委派，允许服务账户访问自身服务

前提：我们已经获得一个服务账户的凭证（hash）或者 TGT

利用流程

1）使用 asktgt 通过获得的凭证 请求一个 TGT，并通过 pass the ticket 的方式将票据注入当前会话

2）将自身账户添加到 自身 msds-allowedtoactonbehalfofotheridentity 属性的 账户列表中

3）通过自身账户的凭证，代表管理员用户执行 S4U，请求自身服务的 TGS

4）通过 pass the ticket 的方式将票据导入当前会话，访问自身服务

备注：如果该账户具有 T2A4D 标志，S4U2Self 返回的 TGS 是可转发的（除非管理员是敏感账户）；但是如果账户没有 T2A4D 标志，那么 S4U2Self 返回的 TGS 是不可转发的，该票据仅能应用于基于资源的受限委派；但是，由于传统受限委派优先判断，那么如果账户 msDS-AllowedToDelegateTo 非空，则 S4U2Proxy。

滥用 基于资源的受限委派，实现域维持访问（类似黄金票据）

如果我们已经已经拿下域控，便可以在域控上配置基于资源的受限委派，允许指定的账户能够访问 krbtgt 服务。

前提：已经获得域管理员凭证

利用流程

1）将账户X 添加到 krbtgt 账户 msds-allowedtoactonbehalfofotheridentity 属性的 账户列表中

2）使用账户X 的凭证，代表管理员执行 S4U，请求 krbtgt 服务的 TGS

3）使用获得的 TGS 执行 asktgs，请求任意服务的 TGS

4）通过 pass the ticket 的方式将票据导入当前会话，访问目标服务

备注：krbtgt 账户的 TGS 等效于 TGT，因为它可以为任意用户生成 TGT。

委派的组合滥用

滥用 基于资源的受限委派，访问另一个配置了 “传统受限委派” 服务所允许访问的服务

在滥用基于资源的受限委派中，我们在执行 S4U2Proxy 后总是会收一个可转发 TGS，所以我们可以利用该 TGS，访问到另一个配置了 “传统受限委派” 服务；并且通过该 TGS 再次执行 S4U2Proxy，请求该服务 msDs-AllowedToDelegateTo 属性中的 SPN/服务 的 TGS。

前提：

1）我们需要能够修改 “受限委派” 服务账户的 msds-allowedtoactonbehalfofotheridentity 属性的值（具备 ACL）；

2）我们已经获得 “受限委派” 服务账户的凭证或者 TGT

利用流程

1）使用 powermad New-MachineAccount 创建一个计算机对象，并设置默认的 SPN

2）将创建的账户添加到 “受限委派” 服务账户 msds-allowedtoactonbehalfofotheridentity 属性的 账户列表中

3）使用新建账户的凭证，代表管理员执行 S4U，请求 “受限委派” 服务的 TGS（可转发）

4）使用 “受限委派” 服务账户的凭证，以及上一步获得的 TGS ，代表管理员执行 S4U2Proxy，请求 “受限委派” 服务账户 msDs-AllowedToDelegateTo 属性 列表中的任意服务 TGS

5）通过 pass the ticket 的方式将票据导入当前会话，访问目标

NTLM 重放 + 基于资源的受限委派，实现 RCE/LPE

这里其实就是在我们没有具备 ACL，不能修改目标对象 msds-allowedtoactonbehalfofotheridentity 属性的情况下，通过 NTLM 重放攻击，将 NTLM 认证信息重放到 LDAP 服务器上，以便对目标配置 “基于资源的受限委派”。

NTLM 重放简介

NTLM 重放就是将获取到的用户 NTLM 认证信息（Net-NTLM Hash）重放到发起主机自身或其他机器上进行认证。

获取 NTLM 认证信息

我们可以通过各种方式来触发计算机账户通过 HTTP 对我们已控机器发起身份验证，在土豆系列中有很多种方式来获取 NTLM 认证信息（详见 ”滥用 NTLM“ ）；另外，我们也可以选择那些可以通过 UNC 路径来触发 NTLM 认证的方式：

内网中的自动身份验证

在内网中，默认情况下，Web 客户端只会对内网中的主机进行自动身份验证，因此主机名中不能出现 ”.“，所以我们设置监听（执行重放）的主机需要在内网中拥有一个合法的 DNS 记录（而不能仅仅只有一个 IP）。

我们可以通过 滥用 ADIDNS 来为我们设置的监听/重放服务器添加一个新的 DNS 记录

第一部分：NTLM 重放攻击部分

我们一般是通过漏洞或其他方式，使某个高权限的账户对我们设置的监听/重放服务器发起 NTLM 认证，我们在获得该账户的 NET-NTLM Hash 后，将其重放到域控制器的 LDAP 服务上，以便让我们具有相应的 ACL 来对目标配置 基于资源的受限委派。举例：

1-1）我们登录 MSSQL 服务，通过调用 xp_dirtree 触发一个到我们恶意 WebDAV 中继服务器的连接，获得 NTLM 认证信息

备注：xp_dirtree ：xp_dirtree 是 MSSQL 中的一个存储过程，它可以使 SQL 服务对提供的文件路径进行目录遍历。

1-2）通过内网中的各种 Web 漏洞或其他方式，触发一个到我们设置的 中继服务器 的连接，获得 NTLM 认证信息

2）我们将获得的 NTLM 认证信息（Net-NTLM Hash）重放到 域控制器的 LDAP 服务上，并配置已控账户到目标的基于资源的受限委派

第二部分：基于资源的受限委派部分

3）我们使用已控账户的凭证来执行 S4U，代表管理员请求到自身服务的 TGS（不可转发），使用该票据请求到目标的 TGS

4）通过 pass the ticket 的方式将票据导入当前会话，访问目标

备注：不同情况下需要具备不同的前提条件，例如，我们要调用 MSSQL 的 xp_dirtree，那么就需要获取该服务的凭证；通过各种 web 漏洞，肯定需要去发现漏洞；执行 S4U 的账户可以提前通过滥用 MachineAccountQuota 创建（或其他方式获取具有 SPN 的账户也行）。

其他绕过技巧

绕过 msDs-AllowedToDelegateTo 白名单限制

KRB_CRED 文件中的服务名称（sname）不受保护，只有服务器名称受保护，这意味着我们可以替换服务名称。因此，如果有两个服务（服务A、服务B）注册在同一个服务账户（账户X）下，而我们仅能访问某个在 msDs-AllowedToDelegateTo 属性中的服务A，却访问不了同样注册在 账户X 下的另一个 服务B，此时，我们可以先请求服务A 的 TGS，然后修改 TGS 中的 服务名称（sname），便可以访问到服务B 了。

结论：我们可以通过修改 TGS 的 sname，来访问注册在同一账户下的其他任意服务。

备注：服务主体名称（SPN）在票据的明文部分，是可以修改的。

绕过敏感账户限制

如果代表的用户是敏感账户或受保护用户组成员，那么在 S4U2Self 处获得的 TGS 是不可转发的，并且没有有效的 SPN，所以使用该 TGS 执行 S4U2Proxy 是失败的；但是，我们可以使用 ASN.1 编辑器修改 S4U2Self 处获得的 TGS ，赋予有效的 SPN，于是该 TGS 可以在 S4U2Proxy 中使用（基于资源的受限委派）。

我们可以先将票据的 Base64 blob 转换为文件，然后使用 ASN.1 编辑器修改
